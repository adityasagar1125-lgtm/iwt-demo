Project Overview 1.1 Introduction The College Club Management System (CCMS) is a full-stack web application designed to centralize the management of student clubs and extracurricular activities within a college ecosystem. The platform serves as a single digital hub where students, club administrators, and college faculty can interact, coordinate events, manage memberships, and communicate effectively. 1.2 Problem Statement Most colleges today manage their club activities through fragmented channels — paper forms, WhatsApp groups, and spreadsheets — leading to poor communication, missed events, and inefficient administration. This project aims to solve these pain points with a structured, technology-driven solution. 1.3 Project Objectives Build a responsive, multi-page web application for managing college clubs Implement RESTful communication between frontend and backend using JSON Use Jakarta Servlets on Apache Tomcat as the backend server layer Use JDBC to connect the backend to a relational database (MySQL/PostgreSQL) Apply CSS Box Model, Flexbox, and Grid layout techniques throughout the UI Implement DOM Manipulation for dynamic user interactions Demonstrate full CRUD operations across all major modules 1.4 Scope The system covers the following major functional areas: Student registration, login, and profile management Club creation, discovery, and joining Event scheduling, registration, and announcement Admin dashboard with club analytics Role-based access control (Student, Club Admin, Super Admin) 2. Functional Requirements 3.1 User Roles Role Permissions & Capabilities Student Register, login, browse clubs, join/leave clubs, register for events, view announcements Club Admin All student permissions + create/edit club info, post events, manage members, post announcements Super Admin Full system access — manage all clubs, users, events, approve/reject club creation requests 3.2 Module-wise Functional Requirements Module 1 — Authentication & User Management FR-01: Students can register with name, email, roll number, and password FR-02: Registered users can log in; invalid credentials show an error via DOM FR-03: Login state stored in sessionStorage; protected pages redirect if unauthenticated FR-04: Users can update their profile (name, bio, profile picture) FR-05: Password change via current password confirmation Module 2 — Club Management FR-06: Students can browse all available clubs with search and filter functionality FR-07: Club cards show name, category, member count, and a Join button FR-08: Students can join or leave a club; membership reflected instantly via DOM update FR-09: Club Admins can create a new club with name, description, category, and logo FR-10: Club detail page shows description, members list, events, and announcements FR-11: Super Admin approves/rejects new club creation requests Module 3 — Event Management FR-12: Club Admins can create events with title, date, time, venue, and description FR-13: Students can browse and register for upcoming events FR-14: Events page displays a filterable grid of upcoming and past events FR-15: Event detail page shows registration count, remaining seats, and countdown timer FR-16: Students receive a JSON-based confirmation response on event registration Module 4 — Announcements FR-17: Club Admins can post announcements with title, body, and priority tag FR-18: Students see a live announcement feed on their dashboard FR-19: Announcements are fetched dynamically via Fetch API and rendered with DOM manipulation Module 5 — Admin Dashboard FR-20: Dashboard shows total clubs, members, events, and recent activity FR-21: Super Admin can deactivate/activate clubs or users FR-22: Club Admin dashboard shows member list with role management FR-23: All dashboard data is fetched from Servlet endpoints as JSON 4. Database Design 5.1 Entity Relationship Overview The database is designed around five core entities: Users, Clubs, Events, Memberships, and Announcements. JDBC PreparedStatements are used throughout for secure, parameterized queries. 5.2 Table Definitions Table: users Column Type Constraint Description user_id INT PK, AUTO Unique user identifier name VARCHAR(100) NOT NULL Full name of the user email VARCHAR(150) UNIQUE Email used for login roll_number VARCHAR(20) UNIQUE College roll number password_hash VARCHAR(255) NOT NULL Hashed password (BCrypt) role ENUM DEFAULT student student / club_admin / super_admin created_at TIMESTAMP DEFAULT NOW() Account creation date Table: clubs Column Type Constraint Description club_id INT PK, AUTO Unique club identifier name VARCHAR(100) NOT NULL Club name description TEXT — Club description and mission category VARCHAR(50) NOT NULL Tech / Cultural / Sports / Academic etc. admin_id INT FK → users Club admin user reference status ENUM DEFAULT pending pending / active / inactive Table: memberships Column Type Constraint Description membership_id INT PK, AUTO Unique membership ID user_id INT FK → users Student who joined club_id INT FK → clubs Club that was joined joined_at TIMESTAMP DEFAULT NOW() Date of joining Table: events Column Type Constraint Description event_id INT PK, AUTO Unique event ID club_id INT FK → clubs Organizing club title VARCHAR(150) NOT NULL Event title event_date DATETIME NOT NULL Date and time of event venue VARCHAR(200) — Event location on campus max_seats INT DEFAULT 100 Maximum registrations allowed 6. API Design — Jakarta Servlet Endpoints All endpoints return JSON responses. Servlets are mapped via @WebServlet annotation on Tomcat 10+. Endpoint Method Description JSON Response /api/auth/register POST Register a new student {"status":"success","userId":12} /api/auth/login POST Authenticate user credentials {"status":"success","role":"student"} /api/clubs GET Fetch all active clubs list {"clubs":[{...},{...}]} /api/clubs/{id} GET Get a single club's details {"club":{...},"members":[...]} /api/clubs/create POST Create a new club (admin only) {"status":"success","clubId":5} /api/clubs/join POST Join a club by clubId {"status":"joined","memberCount":34} /api/events GET Fetch all upcoming events {"events":[{...}]} /api/events/register POST Register student for an event {"status":"registered","seats":12} /api/announcements GET Fetch announcements feed {"items":[{...}]} /api/admin/dashboard GET Get stats for admin panel {"clubs":10,"members":345} 6.1 Sample Servlet Code Pattern @WebServlet("/api/clubs") public class ClubServlet extends HttpServlet { protected void doGet(HttpServletRequest req, HttpServletResponse res) throws IOException { res.setContentType("application/json"); List<Club> clubs = clubDAO.getAllClubs(); res.getWriter().write(new Gson().toJson(clubs)); } } 7. Risk Analysis Risk Impact Mitigation Strategy JDBC connection failures on Tomcat startup High Use connection pooling with singleton DBUtil class, add null checks JSON parsing errors in Servlet response Medium Wrap Gson.toJson() in try-catch and return error JSON on failure Cross-Origin issues in Fetch API calls Medium Add CORS headers in Servlet doGet/doPost using response headers SQL Injection vulnerability High Use JDBC PreparedStatement exclusively — never string-concat queries Session management without Spring Security Medium Use HttpSession API in Servlets for login state tracking Responsive layout breaks on mobile Low Use media queries + fluid grid units; test on Chrome DevTools 11.
